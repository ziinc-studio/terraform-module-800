== Overview

This document captures the exhaustive review of the 800.com REST API
(`https://api.800.com`) and the multi-phase plan to deliver a Terraform
provider that exposes its resources as managed Terraform resources and
data sources.

[NOTE]
====
The user said "Terraform module," but no upstream provider exists for
800.com to wrap. A Terraform _module_ is pure HCL that composes existing
providers; a Terraform _provider_ is the Go binary that talks to a remote
API. Because no provider exists, this repository delivers a *Terraform
provider* (Go, Plugin Framework). HCL convenience modules can be built on
top later in `modules/`.
====

[[discovery]]
== Phase 0 — Discovery (complete)

=== API surface

* Spec: `https://api.800.com/docs/api/public.v2.swagger.yml` (OpenAPI 3.0)
* Vendored copy: `public.v2.swagger.yml` at the repo root.
* *102 paths*, *127 operations*, *16 tags*.
* Base URL: `https://api.800.com`. Other servers in the spec
  (`dev-api`, `stage-api`, `fallback-api`, `localhost`) are 800.com-internal —
  the provider's `endpoint` setting must accept any of them.

=== Authentication

* Bearer token in `Authorization: Bearer <token>`. Tokens are minted from
  *User Settings* in the 800.com web app and are hashed at rest — they
  cannot be retrieved after creation.
* Token format observed: `<userId>|<opaque>` (e.g. `377187|...`).
* Provider configuration takes the token via the `token` attribute,
  the `EIGHT_HUNDRED_API_TOKEN` env var, or both. Env var wins for CI ergonomics.

=== Rate limits

* Spec text claims *60 req/min* on the description.
* *Live observation* (this account, May 2026): `X-RateLimit-Limit: 180`,
  decrementing `X-RateLimit-Remaining` per call.
* The provider must *not* hardcode a number. It must read the
  `X-RateLimit-*` headers and `Retry-After` on `429`, and back off
  per-response.

=== Pagination — three observed shapes

The API was clearly built up in waves. Three distinct envelope shapes
return from list endpoints:

[cols="1,3,3", options="header"]
|===
| Shape | Example endpoint | Cursor fields

| Cursor v2
| `GET /v2/companies/{c}/webhooks`
| `meta.nextCursor`, `meta.prevCursor`

| Page (snake_case)
| `GET /companies/{c}/numberTemplates`
| `meta.current_page`, `meta.last_page`, `links.next`

| Page (camelCase)
| `GET /users`
| `meta.currentPage`, `meta.lastPage`
|===

The HTTP client must abstract over all three so resource code never
sees the inconsistency.

=== Response envelope

Every authenticated call wraps the payload in `{"data": ...}`. Errors are
either `{"errors": [...], "message": "..."}` (validation, auth) or
`{"metadata": [], "data": {"message": "..."}}` (404). The client must
unify both into a typed Go error.

=== Spec drift / known issues

* `GET /v1/users` is documented but returns 404. `GET /users` works. The
  spec is the source of names; live behaviour wins on URLs.
* Several paths use `#fragment` to dedupe operationIds for the same path
  with different `target`/`group` query combos
  (e.g., `/analytics/calls#by-number`). The fragment is for the docs UI
  only — actual requests go to `/analytics/calls?target=…&group=…`.
* OperationIds for older endpoints are MD5 hashes
  (e.g., `62e729339ad731c11a804a79624f36ae`). Newer endpoints have
  human-readable IDs. Generated client code (Phase 4) must rename these.

== Phase 2 — Reference resources & data sources

Pick four endpoints that exercise the full client surface and let us
freeze the conventions before scaling.

* `eight_hundred_webhook` — cursor pagination, simple shape, easy
  destroy. *First resource end-to-end.*
* `eight_hundred_number_template` — page (snake_case) pagination,
  bulk endpoints, validates the second pagination shape.
* `eight_hundred_companies` (data source) — needed by every other
  resource for `default_company_id` lookup.
* `eight_hundred_users` (data source) — page (camelCase) pagination,
  validates the third shape.

Acceptance tests for each, gated by `EIGHT_HUNDRED_API_TOKEN`.

Definition of done:

* `terraform plan` / `apply` / `destroy` round-trip cleanly.
* State drift detection works on out-of-band edits.
* Schema documentation generates without warnings.
* Acceptance tests pass against the live `https://api.800.com`.

== Phase 3 — Core resources

Implement, in order of independence and value:

. `eight_hundred_contact_list` + `eight_hundred_contact_list_membership`
. `eight_hundred_contact`
. `eight_hundred_eav_attribute`
. `eight_hundred_calendar`
. `eight_hundred_calendar_blocked`
. `eight_hundred_calendar_event`
. `eight_hundred_knowledge_document` (with `source` discriminator —
  the three create endpoints map to one resource keyed by `source`,
  which is `ForceNew`).

Each resource lands with: schema, CRUD, import (`terraform import`
syntax: `<company_id>:<resource_id>`), data source twin where it
makes sense, doc page, and acceptance tests.

== Phase 4 — High-value but careful resources

. `eight_hundred_agent` — large schema, skill assignment is a separate
  endpoint and must be modeled as a nested attribute.
. `eight_hundred_call_analytics` (data source) — collapses 13 endpoints
  into one with `target` + `group` enums.
. `eight_hundred_calls`, `eight_hundred_conversations` (data sources).
. `eight_hundred_number` — *paid resource*. Add a required
  `confirm_purchase = true` plan-time check, surface the price in the
  plan output via `PreApply`, and document `prevent_destroy = true` as
  the recommended lifecycle. Destroy releases the number and is *not*
  recoverable.

== Phase 5 — Quality & release

* `tfplugindocs` for full doc tree.
* GoReleaser config to publish signed binaries.
* GPG signing per the Terraform Registry requirements.
* Repository layout and metadata to satisfy
  `https://developer.hashicorp.com/terraform/registry/providers/publishing`.
* GitHub Actions: `lint`, `test`, `acc-test` (manual trigger only —
  acceptance tests cost money once `eight_hundred_number` lands).

== Phase 6 — HCL convenience modules

Once the provider has stable resource coverage, add HCL modules under
`modules/` for common patterns:

* `modules/sms-keyword-autoresponder` — composes
  `eight_hundred_number`, `eight_hundred_webhook`, and a trigger config.
* `modules/contact-list-from-csv` — wraps `eight_hundred_contact` and
  `eight_hundred_contact_list_membership` from a CSV input.

== Risks & open questions

* *Paid actions*: number purchases (`POST /numbers/order`) and
  destroy of `eight_hundred_number` are real-money operations. The
  provider must surface this in the plan. Acceptance tests for the
  Number resource cannot run on PRs — they need a manual workflow.
* *Spec drift*: `/v1/users` 404s in production despite the spec. We need
  a smoke-test job that pings every documented endpoint with a benign
  request to detect drift early.
* *OperationId quality*: many ops have MD5 IDs. If we generate any
  client code from the spec we must alias them to readable names.
* *Rate limit truth*: spec says 60/min, server says 180/min. The client
  reads the headers and ignores the spec.
* *Auth scope*: the test token returned 403 on `/v1/contacts`. The
  provider must map 403 to a clear "missing scope" diagnostic so users
  don't think their token is wrong.
* *Idempotency*: most CREATE endpoints look non-idempotent. If a Create
  succeeds at the API but Terraform fails to record state (e.g. a
  network blip), retry will produce a duplicate. Resources should
  capture the server-assigned ID early and use a *one-shot* retry
  policy on Create — never the standard exponential retry that the
  generic HTTP client uses for reads.
